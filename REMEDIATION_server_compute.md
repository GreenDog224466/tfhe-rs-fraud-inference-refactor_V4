# Remediation — server_compute

**Component:** the blind coprocessor.
**Versions:** V1 (monolith) → V2 (separated pipeline, **baseline**) → V3 (hardened: ZK/CRS/types) → V4 (real-data).
**Baseline for all diffs:** V2 — the first refactor that split encrypt / compute / decrypt into distinct binaries, before any hardening.
**Provenance:** the server changes came from a different path than the client's. A four-model LLM consensus (Gemini 3.1 Pro base; Grok, DeepSeek, Claude reviewers) authored a V2 → V3 server audit/remediation plan. Rather than executing that plan directly, the **plan itself was audited** by Opus 4.7 — an audit of the audit — producing the finalized S1–S18 remediation. Many of those S-items were then deliberately left unimplemented: the decision about which items were actually worth building was made after the finalized plan existed, and that judgment — what *not* to build — is most of what this document records.

The S-numbers below are the plan's own item numbers, kept exactly so each maps to its original meaning. They are therefore not in strict numeric sequence, and several resolve to "decided not to implement" — the decision not to build is itself the recorded judgment.

The V1 → V2 refactor split the binaries and dropped the parallel path; by the V2 baseline the compute is already single-threaded.

## CRS-sharing — discovered, not prescribed (the headline)

**This was not a plan item. It was found by debugging, and it is the strongest single piece of reasoning in the project — the one place a problem was diagnosed from a symptom rather than handed over by an audit.**

**Symptom:** `verify_and_expand` returned `Error { kind: InvalidZkProof }` at V3 bring-up — even with matching keys and correctly reconstructed metadata.

**Root cause:** the client and the server were each calling `CompactPkeCrs::from_config(...)` independently. CRS generation is **probabilistic** — it samples fresh randomness — so the two binaries produced *different* CRSs, and a proof built against one CRS cannot verify against another. The tempting assumption "the keys are deterministic across runs, so the CRS must be too" is **false**: the keys are seeded in genesis; the CRS, generated separately inside each binary, has no shared seed.

**The principle that resolves it:** the CRS (Common Reference String) model is *probabilistic in generation, deterministic in use*. Security requires every prover and verifier to use the **same published string**. The textbook design — and Zama's own guidance — is to generate the CRS once in an offline/setup phase and share it to all parties. Regenerating per-party is simply wrong.

**Fix:** the client generates the CRS once and writes it to `crs.bin`; the server loads the same file. Both sides now use the identical CRS → the proof verifies.

**Design note — CRS generation locus.** The CRS is generated by the client-encrypt binary, written to `crs.bin`, and loaded by server-compute for verification. Generating it in `network_genesis` instead was equally possible — genesis already builds the keys from the same parameter set — and is the production-correct location, since the CRS belongs to the trusted-setup phase rather than to any single participant. The exact rationale for generating it client-side here is not fully recorded; the most plausible reason is that it was the minimal change that resolved the InvalidZkProof failure — the encrypt binary already held the config needed to build the proof, so producing the CRS there and sharing the file was the shortest path to a verifying pipeline. Moving generation into genesis is the natural production refinement.

**Serialization detail (a second sub-problem under the same story):** `CompactPkeCrs` implements none of `Serialize` / `Versionize` / `Named` and has no constructor that reconstructs it from public params (only `new` / `from_config`, which generate fresh), so the CRS object cannot be serialized directly. Both `build_with_proof_packed` and `verify_and_expand` consume `crs.public_params()` (a `CompactPkePublicParams`), so the **public params** are shared instead: compressed via the `Compressible` trait into the `Named` `SerializableCompactPkePublicParams`, `safe_serialize`d with a 1 GB size cap (a memory-exhaustion guard), and `uncompress`ed on the server side before verification.

## Executed — real code changes (S1, S2, S8, S15)

### S1 — Input format: `ProvenCompactCiphertextList` + `verify_and_expand`
**Baseline (V2):** `bincode::deserialize_from` into a raw `Vec<Vec<FheInt64>>`.
**Change:** deserialize a `ProvenCompactCiphertextList`, then `verify_and_expand(&public_params, &public_key, &metadata)`.
**Reason:** the client now emits a proven list, not a vector of vectors — the type must match or deserialization fails outright. The high-level API bundles ZK verification *with* expansion; there is **no plain `.expand()`** on this type. Skipping verification would require dropping to the low-level API. We accept the verification cost for type safety. (In production fhEVM the on-chain precompile may already verify before routing to the coprocessor.)

### S2 — Ciphertext type: FheInt64 → FheInt32 (canonical reasoning)
**Baseline (V2):** `FheInt64` accumulator (16 radix blocks); weights `i64`.
**Change:** `FheInt32` (8 blocks); weights `i32` (the scalar type is forced by the ciphertext type — `FheInt32 * scalar` requires an `i32` scalar).
**Reason:** sizing must be against **accumulation** width, not input width. A single term is 8192 × 75 = 614,400; summed over 433 terms it reaches the millions worst-case. `FheInt16` (ceiling 32,767) silently wraps; `FheInt64` is ~2× the homomorphic cost (HCU) for no benefit at this magnitude. `FheInt32` is the honest minimum.
**Propagation to the client:** this is the decision that overrode the encrypt audits, which had prescribed `FheInt16`. Because the ciphertext type must match on both sides, the server-side analysis here is what drove the client to push `i32`. (See the encrypt remediation, supersession 1.)

### S8 — Feature-count assertion
**Baseline (V2):** none.
**Change:** added `scale_factor: i32` and `feature_count: usize` to the weight struct, and an `assert_eq!(model.weights.len(), model.feature_count, ...)` after the model loads.
**Reason:** a cheap, real, testable integrity check. The JSON declares `feature_count` (433) independently of the weights array; the assertion turns a corrupted or truncated model from a silent wrong answer into a loud, immediate failure *before* any FHE compute runs. Verified: assertion passed (433 == 433), pipeline produced `response.bin`.

### S15 — Byte-for-byte metadata reconstruction
**Baseline (V2):** no metadata (none needed without a proof).
**Change:** reconstruct the exact 44 bytes the client bound into the proof — sender (20×0) + nonce (42) + chain_id (9001) + block_expiry (15,000,000), all big-endian, concatenated in that order.
**Reason:** `verify_and_expand` only succeeds if the server's metadata matches the client's binding exactly. An all-zeros placeholder matches on sender but fails on nonce / chain_id / block_expiry, failing verification. (In production these come from the on-chain transaction context; the mock hardcodes the same values on both sides.)

## Comment-or-skip — deliberately not built (S3–S7, S10–S14, S16)

This is where the judgment lives. Each was an item in the finalized S1–S18 remediation; the decision not to implement it was made afterward, with reasons. The decision not to build is the recorded signal.

### S3 — Keep the ±1 scalar-bypass branches (plan said: delete as dead code)
The plan claimed `weight == 1` (clone) and `weight == -1` (negate) were dead code, assuming no weight equals ±1. **Checked against the weights JSON: false** — there are 10 weights of +1 and 9 of −1 (plus 37 zeros), so 56/433 (~13%) bypass the expensive multiply. The branches fire for real data and are a genuine optimization. Deleting them would remove a real saving and assert something an interviewer could disprove by reading the JSON. **Kept.**

### S4 / S5 — No `OnceLock` caching (model weights, ServerKey)
The plan said cache both in `OnceLock<Arc<...>>`. This binary runs once, scores one request, and exits — it loads each artifact exactly once regardless. Caching saves nothing here, cannot be exercised or tested, and adds complexity. **Comment only**, documenting how a persistent coprocessor would hold them in shared memory at startup.

### S6 / S12 — No worker pool / bounded channel
The plan said build a fixed-size MPSC worker pool with backpressure. A pool distributes work that already exists; this mock has a single file-borne request. Making it real would mean turning the file-in/file-out mock into a networked multi-client server — a systems-engineering signal, not a cryptographic one — and it depends on the S5 caching already declined. **Comment only**, documenting the pool, per-thread key cloning into TLS, and the backpressure rationale (S12).

### S7 / S13 — No chunked accumulation
The plan said rewrite the accumulator into chunks of 16 to enable `batch_bootstrap`. At the high-level API, PBS fires automatically — you cannot control when bootstrapping happens, so chunking is cosmetic here and `batch_bootstrap` does not occur at this layer. The proposed rewrite also dropped the working ±1 branches and rewrote the output-producing loop, risking a silent regression for no functional gain. The carry-budget reasoning (2 carry bits → ~4 additions saturate carry → implicit PBS) is documented (S13); real chunking belongs at the core-crypto level. **Comment only.**

### S10 — Keep the progress print
With a single row the every-4-rows print never fires; it is harmless forward-looking scaffolding for a possible multi-row design. **Kept, no change.**

### S11 — `verify_and_expand` rationale
Documents that there is no plain `.expand()` at this layer, so verification is bundled in — the choice made concrete in S1. **Comment only.**

### S14 — Optimization considered and rejected
For small weights (|w| ≤ 3), repeated addition can sometimes beat scalar multiplication due to PBS overhead. Rejected for this model — the weight distribution [−67, 75] is varied enough that special-casing 2 and 3 adds complexity without meaningful gain. **Comment only**, recorded as depth-of-consideration.

### S16 — Re-randomization variant (MEV)
`verify_re_randomize_and_expand` re-randomizes ciphertexts after verification, breaking mempool linkability between the proven payload and the ciphertext the coprocessor computes on — protection against MEV searchers correlating or replaying. The mock uses the non-rerandomizing variant; production on MEV-heavy chains should prefer the re-randomizing one. **Comment only.**

## Honesty rewrite — S17, S18

Both plan items originally asserted parameters and guarantees the project does not have. They were rewritten to state the true posture, because pasting the original would have been a false claim an interviewer could disprove immediately.

### S17 — Parameter versioning
**Plan said:** pin `..._2M128`.
**Truth:** the project runs `PARAM_MESSAGE_2_CARRY_2_KS_PBS_TUNIFORM_2M64` — the only MESSAGE_2_CARRY_2 TUniform set that exists in tfhe-rs 0.8.7 (the 2M128 constant does not exist at that path; confirmed by compiler error E0425). The honest comment notes that production should pin a *versioned* name (e.g. `V1_0_PARAM_...` in 1.x) to survive upgrades, and that the 1.x migration is scoped as an isolated future pass due to API churn and key-format incompatibility.

### S18 — Security framing
**Plan said:** 2M128, p_fail ≤ 2⁻¹²⁸, IND-CPA^D.
**Truth:** the 2M64 set provides **≥136-bit lattice security** (lattice-estimator: minimum attack `dual_hybrid` ≈136.2 bits; n=887, TUniform(46), q=2⁶⁴) and PBS failure probability ≤ 2⁻⁶⁴. ZK-PoCK via `verify_and_expand` adds integrity binding regardless of p_fail.

**Load-bearing principle (the thing the verdict falls out of):** the correctness margin (p_fail) must match the lattice security level. They are separate axes — noise/dimensions set the ~136-bit security; the 2M64/2M128 suffix sets the PBS failure probability. With ≥136-bit security but only 2⁻⁶⁴ correctness, adversarially-induced decryption failure (2⁻⁶⁴) becomes a far cheaper attack path than breaking the cipher (2¹³⁶) — and on fhEVM a wrong decryption can mean a consensus split or irreversible loss. So 2M64 is fine for honest single-tenant correctness but below the production bar; production needs 2M128 (≤ 2⁻¹²⁸) to align the two axes and reach IND-CPA^D against decryption-oracle attacks. That requires tfhe-rs 1.x, scoped as an isolated final migration so it cannot break the working 2M64 pipeline mid-stream.
